transaction flow
1
Initiation
Student → App
STEP 1.1
Student Opens App
JWT token verified locally. Campus Credit balance and active offers are loaded from backend.
Auth Check
STEP 1.2
Scan Vendor QR
App reads vendor-specific QR. Backend is queried to validate vendor ID and fetch current offers.
Vendor Lookup
STEP 1.3
Order Created
Backend creates an Order record with a unique Order ID, amount, timestamp. Status = PENDING.
DB Write
2
Payment Delegation
App → UPI Intent → GPay / PhonePe
STEP 2.1
UPI Deep Link Fired
App triggers UPI Intent with pa=, pn=, am=, tn= (transaction note) and a unique tr= reference ID.
UPI Intent
STEP 2.2
UPI App Handles Payment
OS hands off to GPay / PhonePe. Student authenticates via UPI PIN. Our app is NOT involved here.
Out of Scope
STEP 2.3
App Receives Callback
UPI Intent returns status (SUCCESS / FAILURE) to our app. ⚠️ Do NOT trust this — it can be spoofed.
Untrusted
3
Server-Side Verification
Razorpay → Backend (S2S Webhook)
STEP 3.1
Webhook Received
Razorpay POSTs a signed payload to /webhooks/payment on our backend. Contains event type and payment data.
S2S POST
STEP 3.2
HMAC Signature Verified
Backend recomputes HMAC-SHA256 of the raw body using the secret key. If hashes don't match → 401 Reject.
🔏 Crypto Check
STEP 3.3
Idempotency Check
Redis lookup: has this payment_id been processed before? If yes → return 200 (ack) but skip processing.
Redis + Dedup
4
Gamification Engine
Backend Atomic Transaction
STEP 4.1
Reward Calculated
Algorithm determines reward: Campus Credits (e.g. 2% of spend), scratch card, or vendor coupon based on spend tier.
Reward Logic
STEP 4.2
Atomic DB Transaction
BEGIN TRANSACTION → update order status → credit Campus Credits → log reward → COMMIT. Rolls back entirely on failure.
⚛ ACID
STEP 4.3
Push Notification Sent
Firebase FCM push sent to student: "Payment successful! You earned 12 Campus Credits 🎉". Vendor notified too.
Firebase FCM
5
Post-Transaction UX
Feedback + Edge Cases
STEP 5.1
Rating Prompt
App shows 1–5 star prompt for the vendor. Saved as a transaction-linked review. Vendor dashboard updated.
Feedback
STEP 5.2
Redeem Credits
Student can apply Campus Credits to reduce payment on next transaction. Deducted atomically same as earning.
Redemption
STEP 5.3
Dispute Flow (Edge Case)
Money deducted but status unclear? Student taps "Report Issue" → backend queries Razorpay API with Order ID → auto-reconciles.
⚠ Dispute API
security layer — your domain
🔐
JWT + TLS
All API calls over HTTPS. Access tokens expire in 15 min. RS256 signing. Refresh token rotation.
Threat: Session Hijacking
🔏
HMAC Verification
Every webhook validated server-side. HMAC-SHA256 computed and compared before any processing begins.
Threat: Webhook Spoofing
♻️
Idempotency + Replay Guard
payment_id stored in Redis. Timestamp tolerance window of ±5 min. Duplicate webhooks silently acked.
Threat: Replay Attacks
🚦
Rate Limiting
Dispute endpoint rate-limited per user. Prevents enumeration attacks on Order IDs via the external aggregator API.
Threat: API Abuse
